Qonduit API Documentation

Qonduit is a Qubic blockchain indexer and RPC server. It provides three API surfaces: a REST API for direct queries, a JSON-RPC 2.0 interface for Bob-compatible and extended methods, and WebSocket subscriptions for real-time streaming.

Interactive API Reference: For a machine-readable OpenAPI spec and interactive try-it-out UI, see the OpenAPI JSON or the Scalar API Reference.

Base URL

http://your-host:8080

All endpoints are served on a single port (default 8080). CORS is enabled for all origins.

Quick Start

Check the server health and get the latest tick:

# Health check
curl http://localhost:8080/health

# Get latest tick data
curl http://localhost:8080/v1/tick

# JSON-RPC: get current tick
curl -X POST http://localhost:8080/rpc \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","method":"getCurrentTickInfo","params":[],"id":1}'

WebSocket Subscriptions

Real-time streaming of blockchain events via WebSocket. Each topic provides a persistent connection that forwards messages from the corresponding NATS subject.

Topics

EndpointNATS SubjectDescription
WS /ws/tickQONDUIT.TICKReal-time tick data as new blocks are finalized
WS /ws/txQONDUIT.TXReal-time transaction data
WS /ws/entityQONDUIT.ENTITYEntity state updates
WS /ws/spectrumQONDUIT.SPECTRUMSpectrum (balance) changes
WS /ws/custom-messageQONDUIT.CUSTMSGCustom messages from the chain
WS /ws/contract-fnQONDUIT.CFNRContract function call results

Connection Protocol

On successful connection, the server sends a JSON confirmation message:

{"status": "connected", "subject": "QONDUIT.TICK"}

After confirmation, the server streams JSON messages as they arrive from the NATS bus. The connection stays open until either the client or server closes it.

WebSocket ping/pong frames are handled automatically by the server.

Optional Query Parameters

ParameterTypeDescription
epochu16Filter by epoch number (optional, reserved for future use)

JavaScript Client Example

// Connect to the tick subscription
const ws = new WebSocket('ws://localhost:8080/ws/tick');

ws.onopen = () => {
  console.log('Connected to Qonduit WebSocket');
};

ws.onmessage = (event) => {
  const data = JSON.parse(event.data);
  console.log('Tick update:', data);
  // Example output: { computor_index: 0, epoch: 142, tick: 14285714, ... }
};

ws.onerror = (err) => {
  console.error('WebSocket error:', err);
};

ws.onclose = () => {
  console.log('WebSocket closed');
};

// To subscribe to transactions instead:
// const wsTx = new WebSocket('ws://localhost:8080/ws/tx');

// To subscribe to entity updates:
// const wsEntity = new WebSocket('ws://localhost:8080/ws/entity');

// Graceful shutdown
process.on('SIGINT', () => {
  ws.close();
  process.exit(0);
});

Pipeline Status

The pipeline_status field in /system-info and getSystemInfo indicates the current state of the data ingestion pipeline.

live Live

The ingestion client is connected to a Qubic node and is caught up (within 100 ticks). Data is being indexed in real-time. This is the normal operating state.

catching_up Catching Up

The ingestion client is connected but more than 100 ticks behind the node. Historical data is being backfilled. Query results may be incomplete.

disconnected Disconnected

No connection to any Qubic node. The ingestion client is offline or has lost its connection. Existing indexed data is still queryable.

query_only Query Only

Ingestion is disabled (query-only mode). The server serves existing indexed data without connecting to a Qubic node. Useful for read replicas or offline analysis.

Status Derivation Logic

if ingestion_disabled:
    pipeline_status = "query_only"
elif not ingestion_connected:
    pipeline_status = "disconnected"
elif ticks_behind > 100:
    pipeline_status = "catching_up"
else:
    pipeline_status = "live"
